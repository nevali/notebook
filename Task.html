Task

Name

task — Submit and manage background processing tasks

Synopsis

    task create [CREATE-OPTIONS] [VAR=VALUE...] [--] COMMAND [ARGUMENTS ...]
    task create [CREATE-OPTIONS] [VAR=VALUE...] -f SCRIPT [ARGUMENTS ...]
    task ls [LIST-OPTIONS] [PATTERN]
    task rm [REMOVE-OPTIONS] PATTERN
    task group create [GROUP-CREATE-OPTIONS] [VAR=VALUE...]
    task group ls [LIST-OPTIONS] [PATTERN]
    task group rm [REMOVE-OPTIONS] PATTERN
    task status [STATUS-OPTIONS] [TASK-OR-GROUP]
    task agent AGENT-OPTIONS
    task runner RUN-OPTIONS
    task server [-]

Description

The task utility allows a user to submit tasks for background processing, list previously-submitted tasks, query their status, and remove them when no longer needed. By default tasks run as the current user on the host the task command is invoked on, however tasks can be submitted and managed on remote hosts via ssh(1), provided the task utility is installed on the remote host, similar to the operation of the rsync utility.

Scope

task is a relatively low-level utility intended for general-purpose background processing with minimal overheads, capable of running on the smallest of modern devices and machine types. It does not on its own have capabilities to schedule tasks across clusters of nodes, configure repeating tasks, integrate with cloud services or hosting providers, nor provide a web-based interactive management interface. That said, it ought, without modification, to be suitable for use as an integral component of a larger system which does provide these kinds of facilities.

task is primarily concerned with the case of a single-user (who may or may not be an administrator) on a single host, with remote orchestration a relatively minor extension.

task does, however, deal with inter-dependencies and external constraints: tasks can be created ad-hoc, or part of a related group which has a shared configuration, common conditions (such as a concurrency limitation), and defined tear-down procedures for both successful and unsuccessful runs. Task groups, which are directed acyclic graphs, can be defined in a single file and launched with one command, or alternatively be launched in a suspended state to be modified on-the-fly and triggered later.

The task processing agent which orchestrates and monitors tasks on an ongoing basis is designed to be as lightweight and unobtrusive as possible, ensuring more resources are dedicated to task processing. Similarly, it does not generally require administrative privileges to run, except when executing tasks which do (or at all, if the task can be executed via sudo(8) configured with an appropriate NOPASSWD option).

Usage

Creating a task

    task create [CREATE-OPTIONS] [VAR=VALUE...] [--] COMMAND [ARGUMENTS ...]
    task create [CREATE-OPTIONS] [VAR=VALUE...] -f SCRIPT [ARGUMENTS ...]

The first form of the command creates an ad-hoc task using the command and arguments specified. Subject to any options specifying starting conditions or other constraints which may apply, the task may be launched by the processing agent immediately.

Operation

Local operation

When a task command is invoked and is not targeting a remote host, the utility first checks to see if the task processing agent, described further below, is running for the current user. If it's not, it's launched in the background, and the task utility waits for confirmation that it has started and a connection can be established to it.

Connections are via a local (Unix domain) socket which resides in a user's home directory, although may be forwarded from a remote host via SSH (see the Remote hosts section below).

Task processing agent

The task processing agent is a background process which periodically scans the queue of waiting tasks, launches any whose starting conditions have been satisfied but have not been started, monitors running tasks for changes of state, and waits for commands from the task utility.

At most one task processing agent runs per user who has tasks queued. The agent terminates itself automatically after a period of inactivity (i.e., no tasks in the queue, and no commands received from the task utility).

Task runner

The task runner is invoked by the task processing agent when it is time to begin processing an individual task—i.e., when its starting conditions have been satisfied. The task runner reads the task details from the queue and spawns the processes required to execute it, monitoring and recording its status until its completion or termination.

Remote hosts

When interacting with a remote host, the task utility attempts to invoke the following command to establish a connection:—

$ ssh [USERNAME@]HOSTNAME task server -

Any prompts from ssh, such as to add an entry to known_hosts or for authentication, will be presented as normal.

If successfully invoked, the task utility on the remote host will launch the background agent process described above if it is not already running, and then forward commands and responses between it and the task utility on the local host via the SSH connection.

Files

The task utility maintains state by storing data in a ~/.task directory (i.e., a directory named .task relative to the user's home directory, as specified by the $HOME environment variable). If this directory is deleted, all task processing state will be lost.

The task processing agent also creates its control socket within the ~/.task directory.

Temporary files, including the working directories for tasks and common storage areas for task groups, are created within one of the locations specified below, listed in order of preference:—

  1. Any locations explicitly specified as part of a task or task group
  2. The location specified by the $TASK_CACHEDIR environment variable
  3. On Darwin systems (macOS, et al), the location returned by $(getconf DARWIN_USER_CACHE_DIR), typically a path within /var/folders.
  4. The location specified by the $TMPDIR environment variable as part of a task or task group itself
  5. The location specified by the $TMPDIR variable in the task processing agent's own environment
  6. /var/tmp, if it is present and new directories can be created within it
  7. /tmp

task is safe to use in scenarios where home directories or cache directories are shared between multiple users or hosts (such as via Ceph or NFS).

Environment variables

The following environment variables are used by the task utility:—

HOME The path to the current user's home directory. A .task directory is created within this path if it does not already exist.

The task runner always initialises a new environment for the task processes it spawns, and will never copy values from its own environment to the child. The following environment variables are set by the task runner and inherited by task processes:—

TASK_ID

TASK_NAME

TASK_START

TASK_GROUP_ID

TASK_GROUP_NAME

TASK_CACHEDIR

In addition to these, any environment variables specified by the task group (if the task is part of a group) will be set, and any environment variables specified by the task itself. By default, when creating an ad-hoc task, any environment variables not beginning with the prefix TASK_ are stored as part of the task details and copied into the environment by the task runner. This behaviour can be inhibited with the -E option to task create, meaning the task is created with an empty environment, plus any variables which are explicitly specified.

Finally, the following variables are set by the task runner to default values if they are not set by the task:—

HOME

LOGNAME

SHELL

PATH

Output and diagnostics

The normal outputs of task create, task ls, task group create, task group ls, and task status are written the standard output stream.

Warning (non-fatal) and error (fatal) messages are written to the standard error stream and in the following format:—

%TASK-W-<CODE>: Warning message describing a condition which has occurred and to which
    the user should be alerted.

    This may be continued over multiple indented lines.

%TASK-E-<CODE>: Error message describing the fatal condition which has occurred
    This may also be continued over multiple indented lines.

    Once an error condition occurs, the task utility will terminate after writing
    the diagnostic message.

<CODE> represents a short alpha-numeric mnemonic identifying the warning or error condition which has occurred to aid in cross-referencing and diagnosis.

Exit status

The following exit values are returned:

0 Successful completion (of the command, not necessarily the task that operations were being performed on).

10 Unable to start the task processing agent

11 Unable to connect to the running task processing agent

12 Failed to read from or write to the ~/.task directory

13 Failed to read from or write to the cache directory

14 The requested sub-command wasn't understood (e.g., task crete ...)

15 The script file could not be accessed

16 The script file could not be parsed

17 No matches for the pattern specified

18 The task ID specified doesn't exist

19 The group ID specified doesn't exist

20 SSH could not be started

21 SSH, or the remote task server, returned a failure response

22 The task processing agent was unable to perform the requested operation

Security considerations

task operates at all times as the user who invokes it; it includes no components which must be executed with elevated privileges and is not configured by default to do so. Whilst it does include a server component, access is via a socket which by default is maintained in a location which is restricted to the (invoking) owning user, subject to filesystem capabilities.

Access from remote hosts rests similarly on the pre-existing ability of users on those hosts to be able to establish an authenticated connection and invoke the task server command.
